Trustless open source file sending
It's 2018, and we at Countable have been unable to find a simple way to send a file to each other or our clients without also sharing it with dozens of cloud provider companies in between. Any of these companies could be next on the news with a data breach. Large risk, or small, there's simply no reason to share anyone's private data with several arbitrary parties. Infact, it's ethically questionable to do so. In many cases we have a non-disclosure agreement with one party, and then send their data through several others with whom we have damage waivers signed when clicking past their terms of service. So, what's the point of an NDA when your data is all over the cloud, and could easily be all over the web in the next breach?

How it Works?
It all happens automatically. Your computer encrypts your data and then we store it. We can't see it because it's encrypted and we don't have the key. Then, you share a link with the intended recipient. The link contains a key to decrypt the data, but [in a way that we never receive it](https://www.w3.org/2001/tag/doc/hash-in-url). All you have to remember is, don't share (or lose) the secure link.

Web Crypto

The Web Crypto API provides a blanket of trustlessness for the cloud. One goal for this project is provide an example for more cloud services to provide client encryption. However, it's not perfect.

The reality is, privacy and security risks are complex, and it's about practical ways to reduce surface area. Web crypto is an additional layer that makes it much harder for an attacker to reach someone's private data. SendCrypt does the following other things to further secure you data:

  • Self hosting (optional) - this way, you need to trust us even less.
  • Open source - to maximize security auditing.
  • SSL - prevents man in the middle javascript injection.
  • No 3rd party code dependencies - reduces exposure to malicious 3rd party javascript.

Just because even combined it's not perfect, doesn't mean users shouldn't have access to the best thing available instead of using file sharing programs with none of these features. Here are some potential risks we havn't solved, take care.

  • Rogue browser extensions - disable extensions you don't trust.
  • Secure link stored in your history - use incognito mode.
  • Secure link is all-powerful - there are no other layers of authentication currently.
  • XSS - We've looked for every possibile exploit we can think of but it's possible we missed one.
Open Source Project
By making CryptSend open source, we allow and encourage peer review to maximize the chances of finding security flaws. This also allows the community involvement and easily allows adding packaging or deployments for various environments.
